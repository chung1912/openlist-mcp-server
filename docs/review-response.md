# 对马老师审查意见的回应（未采纳部分）

## 1. `upload_file` 增加本地路径模式

**暂不增加。**

MCP 工具的输入来自 AI 智能体（Claude 等），不是来自本地文件系统。如果 Claude 要上传文件，它会先通过自己的方式拿到文件内容再调 `upload_file`。加一个"本地路径"参数对 MCP 场景没有意义——Claude 在你的电脑上跑的时候没有权限读任意路径。如果用户确实需要从本地传文件，可以通过 Minis 的 shell 先读文件再转 base64。

## 2. 增加 `openlist-mcp config --print` 子命令

**暂不增加。**

这个 MCP Server 的核心职责是提供文件管理工具，不是配置管理。Claude Desktop 的配置文件路径已经在 README 里写了三个系统的位置，用户按文档操作就够了。加一个子命令反而让入口逻辑变复杂，维护成本高于收益。

## 3. JWT 本地 exp 校验

**暂不增加。**

这个我们同意在安全架构上"更完整"，但实际场景收益很小。OpenList 是用户自己部署的服务，token 过期后客户端有 401 自动重登机制。如果再加上客户端侧的 exp 校验，需要和服务端时钟对齐，引入 NTP 依赖，对于这个工具来说太重了。

## 4. 强制 HTTPS

**暂不强制。**

开发环境、内网环境、Docker 同机部署等场景下 HTTP 是合理的。README 已经明确建议生产环境用 HTTPS，代码层面不做强制，给用户灵活性。

## 5. 明文密码环境变量 / keyring

**暂不支持 keyring。**

环境变量是行业标准做法，Docker、Kubernetes、各种 CI/CD 都这样用。MCP 配置文件加 `chmod 600` 提示已经够用。keyring 支持会让安装和配置复杂度大幅上升，收益和成本不匹配。

## 6. 全局单例线程安全

**暂不加锁。**

asyncio 是单线程模型，`get_client()` 没有多线程竞争风险。如果未来扩展到多线程场景再加锁不迟。

## 7. `httpx.AsyncClient` 退出清理

**暂不处理。**

进程退出时操作系统会回收所有 socket 连接，不存在泄漏问题。MCP Server 是长连接服务，优雅关闭的收益可以忽略。
